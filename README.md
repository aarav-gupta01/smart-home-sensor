# Smart Home Sensor Network

A smart home sensor network using a Raspberry Pi 5 as the main hub and MQTT broker.
An ESP32 microcontroller collects data from a BME680 environmental sensor and an
STHS34PF80 presence sensor, creating a WiFi pipeline from the sensors to the Pi,
which renders the data onto a web dashboard.

## Hardware

| Component | Role |
|---|---|
| Raspberry Pi 5 | Main hub, MQTT broker, dashboard server |
| ESP32 Microcontroller | MQTT publisher, sensor gateway |
| BME680 Environmental Sensor | Temperature, humidity, pressure, VOC |
| STHS34PF80 Presence Sensor | Thermal signature and motion detection |

## Software

The MQTT broker (Mosquitto) is configured and running on the Raspberry Pi.

The ESP32 sensor gateway now reads both sensors over I2C. Firmware is flashed
(MicroPython, ESP32_GENERIC build) and an I2C bus scan confirms both sensors
respond at their expected addresses (`0x5A` for the STHS34PF80, `0x76` for the
BME680).

### AI-assistance disclosure

The register-level driver logic in `esp32/sensors/sths34pf80.py` (register maps,
bit-field definitions, and the embedded-function page access mechanism) was
generated by Codex directly from the STHS34PF80 datasheet, since no MicroPython
port of this sensor exists anywhere. I have studied and can independently explain
the core sequencing methods (`reset()`, `_algorithm_reset()`,
`_write_embedded_function()`, `_safe_set_odr()`) against Section 11 of the
datasheet. The remaining boilerplate and register-lookup portions of the file are
AI-assisted infrastructure that I reviewed and verified against the datasheet, but
did not write from scratch myself.

**Known limitation:** the timing/sequencing of the embedded-function page-switch
procedure references application note AN5867, which I did not have access to
during development or review. That portion of the sequencing is therefore **not
independently verified** against that reference, and should be treated as
provisional until it can be checked.

The `main()` function in `esp32/sensors/sths34pf80.py` was written by me after
studying the sample code in the `Adafruit_CircuitPython_STHS34PF80` GitHub
repository. I used that sample to learn the library's API and typical usage
pattern before writing my own sensor read code; the overall structure follows
that example fairly closely.

### Third-party code attribution

`esp32/sensors/bme680.py` is adapted from
[robert-hh's MicroPython port](https://github.com/robert-hh/BME680-Micropython)
of Adafruit's BME680 driver (originally by Limor "Ladyada" Fried), used under
its MIT license. The `BME680_SPI` class has been removed since this project
only uses the sensor over I2C; a comment in the file marks where it was
removed and why. The `Adafruit_BME680` base class and `BME680_I2C` subclass
are otherwise unmodified. This is standard third-party library use, not
AI-generated code — the license and copyright notice are preserved at the top
of the file. My own integration code (the sensor read logic in `main()`) is
written by me, after studying robert-hh's example usage to learn the driver's
API.

## Project Evolution

This project started with the STHS34PF80 and BME680 sensors wired directly to
the Raspberry Pi 5 for early bring-up and sensor validation. The architecture has
since moved to an ESP32 sensor gateway that reads both sensors over I2C and will
publish readings to the Pi over MQTT. The previous Pi-direct sensor architecture
is preserved in the `pi-only-v1` git tag.

## Current Status

- MQTT broker (Mosquitto) configured on the Pi
- ESP32 flashed with MicroPython; I2C bus validated against both sensors
- STHS34PF80 register-level MicroPython driver written and reviewed (AN5867
  sequencing not independently verified — see disclosure above)
- BME680 soldered and wired to the ESP32; driver (robert-hh port) integrated
  into `esp32/sensors/bme680.py`, `main.py` updated with correct I2C address
  (`0x76`), sensor read logic in progress and not yet tested on hardware
- MQTT publishing from the ESP32 not yet implemented
- Dashboard in planning
